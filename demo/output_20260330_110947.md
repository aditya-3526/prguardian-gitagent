## Blocking Issues

### Blocking Issue 1: Hardcoded Stripe Live API Key

**LOCATION:** `test/fixtures/payment_config.ts:L22`
**OBSERVATION:** A string matching the Stripe live secret key pattern (`sk_live_EXAMPLE_REDACTED_KEY_HERE`) is hardcoded in the test fixture configuration object. Additionally, a Stripe webhook secret (`whsec_test_placeholder`) is present on L23, and the environment is set to `'production'` on L24.
**CONSEQUENCE:** If merged, this key is permanently embedded in the Git history and extractable by any user with repository read access. The `sk_live_` prefix indicates a live (not test) Stripe key. Even if the key value appears redacted, the pattern of committing live-prefixed keys into test fixtures establishes a dangerous precedent and the key format matches Stripe's production key pattern. The `environment: 'production'` setting compounds this — test code is configured to target the production payment environment.
**REQUIRED ACTION:** (1) Remove the hardcoded `stripeApiKey` and `stripeWebhookSecret` values from the test fixture immediately. (2) Replace with environment variable injection (`process.env.STRIPE_TEST_KEY`) or a test mock. (3) Rotate the Stripe API key and webhook secret via the Stripe Dashboard, even if you believe the key is not real — the pattern must not exist in the repository. (4) Change `environment` from `'production'` to `'test'` in the fixture. (5) Add a pre-commit hook or CI check to scan for `sk_live_` and `sk_test_` patterns in committed code.

### Blocking Issue 2: Auth Config Cache with No TTL or Invalidation

**LOCATION:** `src/shared/auth/middleware/validateToken.ts:L4-L8`
**OBSERVATION:** A module-scoped variable `cachedConfig` caches the auth configuration returned by `AuthProvider.getConfig()`. Once set, this cache is never cleared, never refreshed, and has no TTL. The cached value persists for the lifetime of the Node.js process.
**CONSEQUENCE:** If the auth configuration changes at runtime (key rotation, provider update, config push, secret refresh), the cached configuration will be stale. All subsequent token validations will use the old configuration. This creates two failure scenarios: (a) **Stale key rejection** — If auth signing keys are rotated, tokens signed with the new key will be rejected because the validator is using the old config. This causes mass authentication failures. (b) **Security hold-open** — If a compromised key is rotated out, the validator continues accepting tokens signed with the compromised key because the old config is cached. This extends the window of compromise from minutes to the lifetime of the process (potentially days or weeks).
**REQUIRED ACTION:** (1) Add a TTL to the cache (recommended: 60–300 seconds) and refresh the config after expiry. (2) Alternatively, implement an event-driven invalidation mechanism that clears `cachedConfig` when configuration changes are pushed. (3) Add a test that verifies token validation uses fresh configuration after a config change.

## Failure Scenarios

### Scenario 1: "Stale Auth Config — The Silent Lockout"
**Confidence:** HIGH CONFIDENCE
**Trigger:** Auth signing keys or auth provider configuration are rotated (routine security practice). After rotation, `AuthProvider.getConfig()` returns the new configuration. But `validateToken.ts` has already cached the old configuration at module scope. The cached config is never refreshed.
**Impact:** All tokens signed with the new configuration are rejected. Users who authenticate after the key rotation receive tokens that the validator rejects. Estimated impact: 100% of new authentications fail. Existing sessions with old tokens continue to work (paradoxically), creating a split-brain scenario where old users are fine and new users cannot authenticate.
**Detection:** Auth failure rate alert triggers within 5–10 minutes if threshold-based alerting is configured on `401` response rates. Without alerting, first user reports arrive in ~15 minutes.
**Recovery:** Restart all Node.js processes to clear the module-scope cache. This forces `validateToken` to re-fetch configuration. Recovery time: 5–15 minutes depending on deployment rollout speed. No data loss, but significant user friction during the window.

**Postmortem preview:**
> On Friday at 16:52 UTC, PR #247 was merged into main and deployed. The change cached auth configuration at module scope to reduce latency on token validation calls. At 19:30 UTC, the security team executed a scheduled auth key rotation — a routine procedure that occurs monthly. After rotation, new users attempting to log in received 401 errors. The auth failure rate climbed from 0.3% to 67% over 20 minutes. On-call was alerted at 19:43 via PagerDuty. Initial investigation focused on the auth provider, not the application layer, delaying root cause identification by 35 minutes. At 20:18, the on-call engineer identified the module-scope cache and initiated a rolling restart of all API server instances. Service was fully restored by 20:31. Total impact: 1,850 users were unable to authenticate during the 61-minute window. No data loss occurred. Post-incident action: add TTL to auth config cache and add integration test for config rotation scenarios.

### Scenario 2: "Poisoned Promise — The Refresh Cascade Failure"
**Confidence:** HIGH CONFIDENCE
**Trigger:** A token refresh request for User A fails (network glitch, auth provider timeout, rate limit). The failed promise is stored in `pendingRefreshes`. Subsequent refresh requests for User A — including retries — receive the cached rejected promise instead of making a new request.
**Impact:** The affected user is permanently unable to refresh their token for the lifetime of the process. They are effectively locked out of the mobile app until the process restarts or they clear app data and re-authenticate. Under high concurrency, multiple users can be affected simultaneously.
**Detection:** User-reported — "I can't log in" or "the app keeps logging me out." Auth error metrics may not spike if only a small percentage of users are affected at any given time. Detection delay: hours to days.
**Recovery:** Restart affected API instances to clear the `pendingRefreshes` map. Long-term fix: remove completed or failed promises from the map after resolution. Affected users must re-authenticate.

### Scenario 3: "Test Fixture Credential Leak"
**Confidence:** CONFIRMED
**Trigger:** The `sk_live_EXAMPLE_REDACTED_KEY_HERE` value is committed to the repository. Any user with read access to the repository (including CI systems, contractors, open-source forks) can extract it from Git history.
**Impact:** If the key is valid, any entity with the key can process charges, issue refunds, and access Stripe account data. Even if this specific key is not valid, the presence of `sk_live_` prefixed values in the codebase normalizes the practice and increases the probability of a real key being committed in the future.
**Detection:** Automated secret scanning (GitHub push protection, GitGuardian, or similar) may catch this. If no scanner is configured, detection depends on manual discovery — potentially months.
**Recovery:** Rotate the Stripe API key and webhook secret immediately. Scrub the key from Git history using `git filter-branch` or BFG Repo-Cleaner. Notify the security team.
