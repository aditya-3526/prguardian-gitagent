## Blocking Issues

### Blocking Issue 1: Hardcoded Stripe Live API Key

**LOCATION:** `test/fixtures/payment_config.ts:L22`
**OBSERVATION:** A string matching the Stripe live secret key pattern (`sk_live_EXAMPLE_REDACTED_KEY_HERE`) is hardcoded in the test fixture configuration object. Additionally, a Stripe webhook secret (`whsec_test_placeholder`) is present on L23, and the environment is set to `'production'` on L24.
**CONSEQUENCE:** If merged, this key is permanently embedded in the Git history and extractable by any user with repository read access. The `sk_live_` prefix indicates a live (not test) Stripe key. Even if the key value appears redacted, the pattern of committing live-prefixed keys into test fixtures establishes a dangerous precedent and the key format matches Stripe's production key pattern. The `environment: 'production'` setting compounds this — test code is configured to target the production payment environment.
**REQUIRED ACTION:** (1) Remove the hardcoded `stripeApiKey` and `stripeWebhookSecret` values from the test fixture immediately. (2) Replace with environment variable injection (`process.env.STRIPE_TEST_KEY`) or a test mock. (3) Rotate the Stripe API key and webhook secret via the Stripe Dashboard, even if you believe the key is not real — the pattern must not exist in the repository. (4) Change `environment` from `'production'` to `'test'` in the fixture. (5) Add a pre-commit hook or CI check to scan for `sk_live_` and `sk_test_` patterns in committed code.

### Blocking Issue 2: Auth Config Cache with No TTL or Invalidation

**LOCATION:** `src/shared/auth/middleware/validateToken.ts:L4-L8`
**OBSERVATION:** A module-scoped variable `cachedConfig` caches the auth configuration returned by `AuthProvider.getConfig()`. Once set, this cache is never cleared, never refreshed, and has no TTL. The cached value persists for the lifetime of the Node.js process.
**CONSEQUENCE:** If the auth configuration changes at runtime (key rotation, provider update, config push, secret refresh), the cached configuration will be stale. All subsequent token validations will use the old configuration. This creates two failure scenarios: (a) **Stale key rejection** — If auth signing keys are rotated, tokens signed with the new key will be rejected because the validator is using the old config. This causes mass authentication failures. (b) **Security hold-open** — If a compromised key is rotated out, the validator continues accepting tokens signed with the compromised key because the old config is cached. This extends the window of compromise from minutes to the lifetime of the process (potentially days or weeks).
**REQUIRED ACTION:** (1) Add a TTL to the cache (recommended: 60–300 seconds) and refresh the config after expiry. (2) Alternatively, implement an event-driven invalidation mechanism that clears `cachedConfig` when configuration changes are pushed. (3) Add a test that verifies token validation uses fresh configuration after a config change.

## Failure Scenarios

### Scenario 1: "Stale Auth Config — The Silent Lockout"
**Confidence:** HIGH CONFIDENCE
**Trigger:** Auth signing keys or provider configuration are rotated (routine security practice). `AuthProvider.getConfig()` returns new config, but `validateToken.ts` uses the permanently cached old config.
**Impact:** 100% of new authentications fail. Tokens signed with the rotated key are rejected. Existing sessions with old tokens paradoxically continue working — a split-brain scenario.
**Detection:** Auth failure rate alert triggers within 5–10 minutes if threshold-based alerting is configured. Without alerting, first user reports in ~15 minutes.
**Recovery:** Restart all Node.js processes to clear module-scope cache. Recovery time: 5–15 minutes. No data loss.

> **Postmortem Preview:**
> On Friday at 16:52 UTC, PR #247 was merged into main and deployed. The change cached auth configuration at module scope to reduce latency on token validation. At 19:30 UTC, the security team executed a scheduled monthly auth key rotation. After rotation, new users attempting to log in received 401 errors. The auth failure rate climbed from 0.3% to 67% over 20 minutes. On-call was alerted at 19:43 via PagerDuty. Initial investigation focused on the auth provider, delaying root cause identification by 35 minutes. At 20:18, the on-call engineer identified the module-scope cache and initiated a rolling restart. Service was fully restored by 20:31. Total impact: 1,850 users were unable to authenticate during the 61-minute window. No data loss occurred.

### Scenario 2: "Poisoned Promise — The Refresh Cascade Failure"
**Confidence:** HIGH CONFIDENCE
**Trigger:** A token refresh request for User A fails (network timeout, rate limit, provider error). The failed/rejected promise is stored in `pendingRefreshes`. All subsequent refresh attempts for User A return the same rejected promise.
**Impact:** Affected user is permanently unable to refresh their token for the lifetime of the process. Effectively locked out of the mobile app until process restart or app data clear.
**Detection:** User-reported ("I can't log in"). May not trigger alerts if only a small percentage of users affected at any time. Detection delay: hours to days.
**Recovery:** Restart affected API instances to clear `pendingRefreshes` map. Long-term fix: add `.finally(() => this.pendingRefreshes.delete(userId))` after promise resolution.

### Scenario 3: "Credential Leak via Test Fixture"
**Confidence:** CONFIRMED
**Trigger:** `sk_live_EXAMPLE_REDACTED_KEY_HERE` is committed to repository. Extractable from Git history by anyone with read access.
**Impact:** If the key is valid — unauthorized Stripe charges, refunds, and account data access. If not valid — normalizes the pattern of committing `sk_live_` prefixed values, increasing future leak probability.
**Detection:** GitHub push protection or external secret scanners may catch it. Without automated scanning, discovery depends on manual review — potentially months.
**Recovery:** Rotate Stripe API key and webhook secret immediately. Scrub from Git history with `git filter-branch` or BFG Repo-Cleaner.
